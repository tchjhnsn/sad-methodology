# Contributing to SAD

SAD is a solo-authored methodology maintained by Torian Johnson. Contributions are welcome within a narrow scope; the methodology's coherence depends on a small, consistent authorial voice.

## What's welcome as a Pull Request

- Typo fixes.
- Clarifications of existing language where ambiguity or imprecision is evident.
- Corrections to outdated references, broken links, or formatting.
- Translations of existing content into other languages.
- Examples or case studies from your own adoption of SAD that illustrate an existing concept (as a separate file in the repo, referenced from README or the walkthrough).

## What's better raised as an Issue first

- Proposed structural changes to the methodology (new phases, removed phases, altered Convergence thresholds).
- Challenges to the methodology's core claims or premises.
- Questions about adopting SAD in a domain it doesn't cover yet (Brand, Business, physical products, services, content).
- Suggestions for new capability-skills or changes to existing ones.
- Anything that would change the interpretation of the methodology rather than clarify existing interpretation.

These are discussion material, not merge material. Open an issue, and I'll engage there before any PR is warranted.

## What's out of scope

- Tooling, CLIs, generators, or frameworks built around SAD. Those are welcome as *separate repos* that reference SAD; they don't live here. SAD is documentation, not software.
- Applying SAD to non-software domains in this repo. The methodology-family roadmap tracks variants for Brand, Business, and other product types; those will live in separate repos when authored.
- Renaming or restructuring the methodology. Structural decisions are authorial.

## Code of conduct

Standard open-source expectations: engage in good faith, assume good intent, attack ideas not people, be patient with reviewers. The repo is small enough that formal code-of-conduct enforcement is lightweight — if something becomes a problem, I'll address it directly.
